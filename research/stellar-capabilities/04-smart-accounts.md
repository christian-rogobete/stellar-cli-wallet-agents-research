# Smart Accounts (Account Abstraction)

**Status:** complete
**Last updated:** 2026-04-18
**ID:** 04-smart-accounts

---

## 1. What it is

A **smart account** (C-account) is a Soroban contract implementing `CustomAccountInterface`: the host calls `__check_auth(signature_payload, signatures, auth_contexts)` per signature and the contract decides whether to accept, running arbitrary Rust — caps, allowlists, session windows, passkey verification, quorum checks — before returning `Ok(())`. Where a G-account is a fixed k-of-n multisig, a C-account is a general-purpose, upgradeable policy engine at the auth boundary [`developers.stellar.org/.../smart-wallets` @ 2026-04-18; local `multisig-smart-account/account/src/contract.rs:72-80 @ 3f81125`]. For an agent wallet this is the **on-chain policy backstop** that survives host compromise, injected tool calls, and sub-agents on remote hosts because enforcement lives in the ledger.

## 2. Components

- **`CustomAccountInterface::__check_auth`** — host entry; one `Context` per `require_auth` [local `multisig-smart-account/account/src/contract.rs:72-80 @ 3f81125`].
- **OpenZeppelin `stellar-accounts` v0.7.1** (MIT) — reference framework; traits `SmartAccount`, `Policy`, `Verifier`, `ExecutionEntryPoint` [local `packages/accounts/src/smart_account/mod.rs:136-515 @ 3f81125`].
- **`ContextRule`** — auth entry typed by `ContextRuleType` (`Default`, `CallContract(Address)`, `CreateContract(BytesN<32>)`); holds signer IDs, policy IDs, optional `valid_until`; max 15 signers / 5 policies per rule, unlimited rules per account [local `storage.rs:140-150, 632 @ 3f81125`].
- **`Signer`** — `Delegated(Address)` or `External(Address, Bytes)` (verifier contract + key) [local `storage.rs:94-102 @ 3f81125`].
- **`Verifier` trait** — pure verifier contracts; shipped: ed25519, secp256r1 WebAuthn [local `verifiers/webauthn.rs @ 3f81125`].
- **`Policy` trait** — `install / enforce / uninstall`; shipped: `simple_threshold`, `weighted_threshold`, `spending_limit` (rolling window) [local `policies/mod.rs:47-162 @ 3f81125`; `policies/spending_limit.rs:85-104 @ 3f81125`].
- **`AuthPayload { signers, context_rule_ids }`** — one rule ID per context; signers authenticate over `sha256(signature_payload || context_rule_ids.to_xdr())` to close a rule-downgrade hole [local `storage.rs:103-138 @ 3f81125`].
- **`ExecutionEntryPoint::execute`** — account-scoped call, mainly for policies to re-enter their owner; **not** a Safe-style `execTransactionFromModule` [local `smart_account/mod.rs:494-515 @ 3f81125`].
- **`Upgradeable` trait** — `upgrade(wasm_hash, operator)` gated only by `require_auth`; no built-in timelock, no migration trait [local `contract-utils/src/upgradeable/mod.rs:1-120 @ 3f81125`].
- **Passkey Kit / Smart Account Kit** — client SDKs assembling `AuthPayload`, driving WebAuthn, submitting via optional OZ Relayer. Two current kits share design (manager split, default deployer seed, wire-compatible signer formats): kalepail's TypeScript **SAK** for browser targets [local `smart-account-kit/README.md @ e731a3d`] and Soneso's **KMP `OZSmartAccountKit`** for Android / iOS / macOS / Web (part of `kmp-stellar-sdk`) [local `kmp-stellar-sdk/docs/smart-accounts/README.md @ c7bfedb`]. `passkey-kit` is the legacy pre-OZ reference [local `passkey-kit/README.md @ 1b3aa63`]. See `research/external/stellar-ecosystem/smart-account-kit.md` for the side-by-side analysis and N2/N3 implications of the default hosted indexer.
- **Classic fee-bump** — any smart-account-authorised inner tx can be wrapped by a G-account `FeeBumpTransaction`; Soroban auth-entry signing lets fee payer and authoriser differ [`developers.stellar.org/.../fee-bump-transactions` @ 2026-04-18].

## 3. Relevance to an AI-agent wallet

- **Actors:** A1, **A2**, A3, **A4**, A5.
- **Non-negotiables:** N1, N2, N3, N6.
- **Threats mitigated:** **T2** (policy in ledger blocks injected drafts exceeding caps), **T4** (per-sub-agent rule with `valid_until` is on-chain revocable), **T6** (`spending_limit` rolling window), T10 (wasm is network-agnostic), T11 (auth-digest rule-ID binding closes rule-downgrade by sponsor). **Created:** **T8** (policy-wasm supply chain), T12 (signer-set divergence), T13 (mutable policy / verifier addresses), T14 (cross-account verifier compromise), T1 expansion (upgrade-admin key).

Load-bearing for the CLI-vs-agent-native decision: option B can treat the smart account as the authoritative policy surface; option A uses it as belt-and-suspenders behind the local engine. Either way the choice depends on what the account enforces today versus what we must add (§7).

**Canonical AI-agent template (OZ README §"Use Cases" §3).** The OZ `stellar-accounts` README names AI Agents as an explicit use case with a worked example: `context_type: Default` (or a scoped `CallContract(address)`), the agent's key as the signer, and two policies — a whitelist policy for allowed functions, plus a balance / spending-limit policy. This is the actor-model's A2 (multi-agent orchestrator) mapped directly onto OZ primitives. Sub-agent delegation in A2 is implemented as **one context rule per sub-agent** rather than many signers on a shared rule — OZ's 15-signer-per-rule hard cap makes per-sub-agent rules the correct architectural choice anyway, and it preserves the isolation property that revoking one sub-agent only affects that sub-agent's rule. The Session Logins and Backend Automation use cases in the same README are closely related patterns for A4 and A5. A wallet implementation could surface these three templates as named starting configurations.

## 4. Agent-specific quirks

- **Caller pre-selects the rule.** `context_rule_ids` is mandatory, index-aligned with `auth_contexts`, no auto-discovery. The client keeps an intent → rule-ID map in sync with on-chain state; length mismatch returns `ContextRuleIdsLengthMismatch = 3014` [local `storage.rs:103-138 @ 3f81125`].
- **Signer signs the auth digest, not the payload.** `sha256(signature_payload || context_rule_ids.to_xdr())`. Direct-XDR agents that skip this produce valid-looking signatures that fail `__check_auth` [README §"Auth digest now binds `context_rule_ids`" @ 3f81125].
- **Simulation misses `Delegated` auth entries.** OZ README: `Signer::Delegated` "requires manual authorization entry crafting, because it is not returned in a simulation mode" [local `packages/accounts/README.md:96 @ 3f81125`]. This is an extra client-side step, not a blocker — a wallet can use `Delegated` signers for agents as long as its client code constructs the `AuthPayload` auth entries explicitly rather than relying on simulation to surface them. Preflight-reliant agent harnesses that cannot add the manual-crafting step should prefer `Signer::External` + verifier.
- **Silent threshold drift.** Adding a signer weakens 3-of-3 to 3-of-4 without error; removing can strand the rule. Signer edits must bundle threshold updates via `ExecutionEntryPoint` [README "Signer Set Divergence" @ 3f81125].
- **TTL.** Rule storage is persistent with a 30-day extend bumped on each auth check [local `smart_account/mod.rs:519-521 @ 3f81125`]. Idle sub-agent rules can expire and need restore — a footgun for A1/A5.
- **No `x402` wiring.** Nothing in OZ, Passkey Kit, or Meridian Pay ships a pay-per-resource policy; the closest primitive is a session-key + per-counterparty `CallContract(addr)` rule.
- **Deterministic contract address across SDKs.** SAK and KMP `OZSmartAccountKit` both derive their default deployer from `SHA256("openzeppelin-smart-account-kit")`, which means identical credential IDs produce identical contract addresses regardless of which SDK deployed the account (subject to byte-identical credential-ID encoding — see `research/external/stellar-ecosystem/smart-account-kit.md` §11). A wallet ships the same derivation helper and can discover a user's contract address without an indexer by re-deriving it from the credential. This is strong interop for a companion-UI story that spans web (SAK) and native (KMP) surfaces [SAK `src/kit.ts @ e731a3d`; KMP `OZSmartAccountConfig.kt @ c7bfedb`].
- **Active-rule enumeration: on-chain scan works; indexer is a convenience.** The on-chain contract exposes `get_context_rule(id)` and `get_context_rules_count()` but not an iterator over active rule IDs after deletions. **KMP's `OZContextRuleManager.getAllContextRules()` does the scan natively** — loops `[0, count)` on-chain and filters deleted entries client-side; no indexer involved. **SAK's `rules.list()` / `rules.getAll()` defaults to a hosted indexer** (`smart-account-indexer.sdf-ecosystem.workers.dev`, Cloudflare Worker with Goldsky pipeline). The indexer reads only public on-chain events, so this is a liveness and query-pattern concern, not a data-exposure one. A self-custodial wallet using SAK for rule enumeration either re-implements the scan pattern atop `get_context_rule(id)` + `get_context_rules_count()` or accepts the dependency; credential-to-contract reverse lookup remains indexer-backed in both SDKs unless the caller uses deterministic-address derivation.

## 5. Status on the network

- **Mainnet / Testnet.** Production on both; current protocol 25 ("X-Ray", live 2026-01-22). Custom `__check_auth` since Protocol 20 (2024-02-20); secp256r1 (WebAuthn) since Protocol 21. OZ `stellar-accounts` deployed by real users via `smart-account-kit` and Meridian Pay's own account [`developers.stellar.org/docs/networks/software-versions` @ 2026-04-18; `stellar.org/blog/developers/protocol-20-and-smart-contracts-are-live-on-mainnet` @ 2026-04-18].
- **OZ library maturity.** `stellar-accounts` 0.7.1, MIT, with audit reports for every tag 0.1.0-RC through v0.7.0 [[`stellar-contracts/audits/ @ 3f81125`](https://github.com/OpenZeppelin/stellar-contracts/blob/main/audits/ @ 3f81125)]. README warns "rapid iterations are expected" [local `packages/accounts/README.md @ 3f81125`]. Breaking changes v0.6 → v0.7.0 (consolidated from the OZ migration guide): (1) `AuthPayload` changed from tuple struct to named struct with mandatory `context_rule_ids: Vec<u32>`; (2) auth digest now binds `context_rule_ids` via `sha256(signature_payload ‖ context_rule_ids.to_xdr())` — off-chain signers must compute this digest, not sign the raw payload; (3) `Policy` trait's `can_enforce()` method removed, `enforce()` now handles both validation and state mutation; (4) automatic newest-first rule iteration removed — callers must explicitly supply `context_rule_ids`; (5) `get_context_rules` (deprecated vec-returning method) removed in favour of `get_context_rule` + `get_context_rules_count`; (6) `TooManyContextRules` error renamed to `MathOverflow` (there is no upper limit on rules per account; error code 3012 now means an internal ID counter hit `u32::MAX`); (7) `remove_signer` and `remove_policy` now take `u32` IDs instead of full `Signer` / `Address` objects (IDs are emitted in `signer_added` / `policy_added` events); (8) global signer/policy registry with dedup by XDR encoding (behavioural change — same signer object is stored once regardless of rule count, with reference counting).
- **Roadmap.** Protocol 23 (2025-09-03) cheapens cross-contract calls and adds `get_executable`, which OZ leans on for multi-policy composition and wasm-hash-pinned policy trust [`stellar.org/blog/developers/announcing-protocol-23` @ 2026-04-18; `docs.openzeppelin.com/stellar-contracts/accounts/smart-account` @ 2026-04-18].
- **Incidents.** The 2024 fee-bump bug disclosure reshaped sponsor guidance: build the inner tx on the sponsor from user-signed auth payloads + invocation intent, not full XDR [`stellar.org/blog/developers/fee-bump-bug-disclosure` @ 2026-04-18].

## 7. Known gaps or pain points

- **Safe-style module-initiated execution is absent; `ExecutionEntryPoint` is the closest primitive but a narrower one.** Policies in OZ v0.7.1 only `enforce()` during `__check_auth`; they cannot originate a tx [local `policies/mod.rs:47-162 @ 3f81125`; cf. `research/external/crypto/safe.md` §11]. `ExecutionEntryPoint::execute(target, target_fn, target_args)` lets the account invoke any target contract as the authoriser (the README highlights the policy-update case — `set_threshold()` and similar — but the trait itself is general-purpose and works for any contract the account wants to call as self-auth). It is explicitly *not* a Safe-style `execTransactionFromModule` where a *policy* originates an outbound transaction: the account signs, not the policy. The gap is in the policy-origination direction, not in the account-invokes-anything-as-self direction.
- **No policy-attestation registry.** Safe has ERC-7484 (Rhinestone); OZ has nothing equivalent. Installing a policy means trusting the deployer by address [safe.md §9, §12.12].
- **No `Guard` trait.** Safe `checkTransaction` gates every tx globally; OZ policies are rule-scoped, so a "never above M per period regardless of rule" invariant must either be replicated per-rule or implemented by attaching a `spending_limit` policy to a `Default`-context rule that matches every operation [safe.md §11]. The latter approach covers most real cases but is not architecturally equivalent to a global guard.
- **Selector-level context typing missing; amount-level scoping is provided.** `CallContract(Address)` is contract-level only — expressing "only the `transfer` function on that contract" requires a custom policy parsing `Context::Contract { fn_name, args }`. The shipped `spending_limit` policy *does* provide arg-level scoping for amount-based rules, so not every arg-level constraint is a gap; the Passkey Kit sample shows the selector-parsing shape but is trivial and unaudited [local `passkey-kit/contracts/sample-policy/src/lib.rs @ 1b3aa63`].
- **Fee-payer indirection.** No ERC-4337 paymasters. Fees come from a G-account `sourceAccount` or a `FeeBumpTransaction` source — both must be classic accounts. A C-account paying directly for another C-account's invocation is unsupported [`developers.stellar.org/.../fee-bump-transactions`].
- **LaunchTube / OZ Relayer dependency for passkey flows.** Passkey-only sub-agents sign only auth entries; a classic fee source is still needed. SDF's answer is LaunchTube → OZ Relayer Channels Plugin — licence and self-hostability are an open N4 question [`analysis/00-context.md` §5.4].
- **`Delegated` signers require manual auth-entry crafting** (see §4 — not a blocker, an extra client-side step).
- **Upgrade timelock not wired by default.** `Upgradeable::upgrade` is gated only by `require_auth` [local `contract-utils/src/upgradeable/mod.rs:1-120 @ 3f81125`]. `stellar-governance::timelock` is available as a separate crate in the same OZ `stellar-contracts` family and can be composed in, but OZ does not wire it into `stellar-accounts` by default — wallets wanting upgrade delay have to ship the composition themselves.
- **Global signer / policy registry with dedup (v0.7 behavioural change).** Signers and policies are stored in a global per-account registry with reference counting; the same signer object (identified by its XDR encoding) is stored once regardless of how many rules reference it, and storage is cleaned up automatically when the reference count reaches zero. For a wallet deploying many per-sub-agent rules with a shared spending-limit policy, this keeps storage costs linear in unique signers/policies rather than total references. Wallet UX should surface the shared-ness explicitly so removing a signer from one rule does not mislead the user into thinking it is removed globally — other rules may still reference the same signer ID [OZ README §"Caveats" item on global registry, §"Migration Guide: v0.6 → v0.7" item "Global signer and policy registry"].
- **Smart account vs. classic multisig.** G-account thresholds are free, no TTL, hardware-wallet native, but cannot express caps or session windows. C-accounts are strictly more expressive at the cost of deploy, TTL, and policy-wasm supply chain. Support both: A5/A7 usually need only classic multisig; A1/A2/A4 need a smart account.
- **No MPP-to-smart-account integration in the reference SDK.** `@stellar/mpp` v0.5.0 assumes single-keypair signing (`resolveKeypair` + `envelopeSigner`). Credential-construction paths for multisig, passkey-on-C-account, or policy-signer accounts are not first-class. Architecturally, the composition is natural — agentic-payment protocols that sign Soroban auth entries (x402, MPP charge) route through `__check_auth` on a C-account `from`, which invokes the account's context-rule / signer / policy tree. The gap is SDK wiring, not architecture. See `10-mpp.md` §7 item 8 for the MPP-specific construction path.

## 8. Candidate requirements surfaced

- `REQ-sa-oz-baseline`: **MUST** support OZ `stellar-accounts` pinned at exact version `= 0.7.1` (not `v0.7+`; pre-1.0 library with expected breaking changes). `AuthPayload` carries `context_rule_ids: Vec<u32>` with one ID per auth context (no auto-iteration). Off-chain signing MUST compute the auth digest as `sha256(signature_payload ‖ context_rule_ids.to_xdr())` and sign that digest — this is the v0.7 hardening that closes rule-ID downgrade (T11). Version bumps are release events requiring migration review. §2, §4, §5 breaking-changes list.
- `REQ-sa-passkey-signer`: MUST act as an `External` secp256r1 signer against a WebAuthn verifier. §2, §5.
- `REQ-sa-session-rule`: MUST provision per-sub-agent `ContextRule`s with `valid_until`, scoped signers, and a spending-limit policy; revocation MUST take effect on-ledger for newly-constructed signatures before the next submission (already-signed envelopes in-flight are a separate concern; see `02-threat-model.md` T4 and `01-actor-model.md` A2). §3, §7.
- `REQ-sa-policy-selector-scope`: NICE. Selector-level context typing is a gap vs Safe modules; `spending_limit` policy already provides amount-level arg scoping. A project that needs function-selector scoping ships a custom policy parsing `Context::Contract { fn_name, args }`. §7.
- `REQ-sa-guard-equivalent`: SHOULD provide a "never above M per period regardless of rule" invariant. OZ-shaped substitute: attach a `spending_limit` policy to a `Default`-context rule. A Safe-style global `Guard` trait is not available in OZ v0.7.1 and is not on the upstream roadmap. §7.
- `REQ-sa-upgrade-timelock`: SHOULD. A deployed smart account should not be upgradeable without a user-approved minimum delay (`stellar-governance::timelock` composed in) or an explicit "admin = burn" post-deployment. OZ does not wire a timelock by default. §7.
- `REQ-sa-fee-payer-parent`: MUST support submitting a smart-account-authorised inner tx via a parent G-account `feeSource`, including `FeeBumpTransaction`, without the smart account signing the fee payment. §2, §7.
- `REQ-sa-no-relayer-hard-dep`: Any passkey smart-account flow MUST be submittable against a vanilla RPC + user-held G-account fee payer; LaunchTube / OZ Relayer MUST be optional (N2, N3). §7.
- `REQ-sa-delegated-signer-warn`: MUST warn when a user configures `Signer::Delegated` on an agent-serving account that relies on simulation for auth-entry discovery; recommend either adding the manual auth-entry crafting step in the client, or using `Signer::External` + verifier for preflight-reliant flows (T3). §4.
- `REQ-sa-attestation-optional`: SHOULD pin and verify a wasm hash for any policy contract installed on an account, surfacing the hash in the approval UI (T8). §7.
- `REQ-sa-verifier-pinning`: **MUST** pin verifier contracts by wasm hash at rule-install time (parallel to `REQ-sor-policy-contract-pinning` for policy contracts); refuse install at mutable / upgradeable deployment addresses unless user opts in; surface pinned hash in approval UI. Defends against T13 (policy/verifier address replacement) and T14 (cross-account verifier compromise). §7, §2 component list.
- `REQ-sa-atomic-signer-threshold-update`: **MUST** bundle signer add/remove with threshold/weight updates into a single `ExecutionEntryPoint` transaction; UI refuses signer removal that would strand an attached threshold. Defends against T12 (signer-set-divergence brick). §4, §7.
- `REQ-sa-context-rule-caps`: SHOULD. UI surfaces OZ's per-rule hard limits (≤15 signers, ≤5 policies) and enforces the one-rule-per-sub-agent pattern for A2 orchestrators. §2 component list, §3 (A2 mapping).
- `REQ-sa-classic-multisig-parity`: MUST treat classic G-account multisig as first-class for A5/A7; MUST NOT force smart-account deployment when classic multisig covers the case. §7.
- `REQ-sa-agent-usecase-example`: SHOULD ship an "AI agent" `ContextRule` template (signer + allowlist + balance-% policy) mirroring OZ README §"Use Cases" §3. Promoted from NICE to SHOULD given A2 centrality in the agent-wallet actor set. §3.

### Gap list vs. Safe modules

- Module-initiated (policy-originated) execution: missing from OZ v0.7.1; policies gate, they do not drive; `ExecutionEntryPoint` covers only the narrower "account-calls-its-own-policy-update" direction, not the Safe `execTransactionFromModule` direction [safe.md §11; OZ README §"ExecutionEntryPoint"].
- Module / policy attestation registry: missing; no ERC-7484 analogue; needs a registry contract keyed by wasm hash [safe.md §9, §12.12].
- Global transaction guard (Safe `checkTransaction`): missing; OZ policies are rule-scoped. A `spending_limit` policy on a `Default`-context rule provides a close substitute for amount-based global invariants but is not architecturally equivalent to Safe's guard [safe.md §11].
- Role-based target + selector + arg scoping: partial. `CallContract(Address)` is contract-level only; the shipped `spending_limit` policy covers amount-level arg scoping, but function-selector scoping requires a custom policy parsing `Context::Contract { fn_name, args }` [safe.md §11].
- Management lifecycle events: **present** — OZ v0.7.1 emits `context_rule_added`, `context_rule_meta_updated`, `context_rule_removed`, `signer_added`, `signer_removed`, `policy_added`, `policy_removed` [local `src/smart_account/mod.rs:222-466 @ 3f81125`]. Events for policy-*originated* execution are absent by design — OZ policies cannot originate calls, so there is nothing to emit (this is a consequence of the first gap above, not a separate gap).
